# Phase 5.S3 ŌĆö Effort / reasoning parameter feasibility spike

> Issue: [#64](https://github.com/Cybourgeoisie/openrouter-agent-coder/issues/64) ŌĆö investigation only, no `src/` changes.

**Question**: Does the OpenRouter API (and `@openrouter/agent` SDK) accept a `reasoning` / `effort` / `thinking_level` parameter, and if so, what's its shape (enum? number? per-provider passthrough?)?

**Verdict**: **Yes (Full).** Both OR endpoints accept a normalized `reasoning` object with an `effort` enum (`xhigh | high | medium | low | minimal | none`); on `/responses` (which `callModel` targets) the object additionally accepts `enabled`, `max_tokens`, and `summary`. The `@openrouter/agent` `callModel` already exposes the field through its `CallModelInput` type ŌĆö no SDK lift needed; Card 5.4 is a pure wire-up from our existing `OpenRouterAgentRunOptions.effort` stub into the `callModel` call.

**Date**: 2026-05-24
**Spike author**: Claude (delegated by Phase 5.S3 issue #64)
**Refines**: Phase 5.4 (Effort / reasoning level pass-through)

## TL;DR

- OR has a **normalized** `reasoning` parameter, not a per-provider passthrough. OR maps the requested `effort` to each provider's native shape (OpenAI `reasoning_effort`, Anthropic `thinking.budget_tokens`, Gemini `thinkingLevel`, Qwen `thinking_budget`).
- `effort` enum: `xhigh | high | medium | low | minimal | none` (typed as `OpenEnum` in the SDK ŌĆö additional string values are accepted by the wire but not type-suggested). If a model doesn't support a requested level, OR maps to the nearest supported level.
- `/responses` (what `callModel` targets) accepts the richer `ReasoningConfig` (`effort`, `summary`, `enabled`, `maxTokens`); `/chat/completions` accepts a slimmer `Reasoning` (`effort`, `summary` only). We're on `/responses`, so we get the full shape.
- **No agent-SDK lift required.** `@openrouter/agent@0.3.2`'s `CallModelInput<TTools>` extends `Omit<ResponsesRequest, 'stream' | 'tools' | 'input'>`, so `reasoning?: ReasoningConfig` is already a first-class field on `callModel`. We just pass it through.
- **Card 5.4 refined estimate: 3ŌĆō5h** (was `~5h passthrough`). Net-down from the placeholder because the SDK already accepts the field ŌĆö we just plumb our existing `OpenRouterAgentRunOptions.effort` stub into `agent.ts:912`'s `callModel({...})` and tighten the option type from `string` to the documented enum.

## OR API findings

### Documented parameter

Source: [OpenRouter ŌĆö Reasoning Tokens](https://openrouter.ai/docs/guides/best-practices/reasoning-tokens) (observed 2026-05-24).

```json
{
  "reasoning": {
    "effort": "high",
    "max_tokens": 2000,
    "exclude": false,
    "enabled": true,
    "summary": "auto"
  }
}
```

| Sub-field    | Type                                                            | Notes                                                                                                                                |
| ------------ | --------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| `effort`     | `"xhigh" \| "high" \| "medium" \| "low" \| "minimal" \| "none"` | Normalized enum. Allocates a percentage of tokens for reasoning (~95% xhigh down to ~10% minimal per docs).                          |
| `max_tokens` | int                                                             | Exact reasoning-token budget. Best-supported on Anthropic + Gemini. (1024ŌĆō128000 documented range for Anthropic.)                    |
| `exclude`    | boolean                                                         | Run reasoning silently ŌĆö model uses it but it's omitted from the response payload.                                                   |
| `enabled`    | boolean                                                         | Activate reasoning with provider defaults (use when neither `effort` nor `max_tokens` set).                                          |
| `summary`    | `"auto" \| "concise" \| "detailed"`                             | Verbosity of the surfaced reasoning summary. (Wire enum; values cross-checked against `ReasoningSummaryVerbosity` in the typed SDK.) |

> Behavior when a model doesn't support a level: "OpenRouter will map your requested effort to the nearest supported level." Per-model support is a wire concern, not a caller concern.

### Per-endpoint coverage

| Endpoint                                     | `reasoning` sub-fields accepted                                 | Source                                                                                              |
| -------------------------------------------- | --------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| `POST /api/v1/chat/completions`              | `effort`, `summary` only (`Reasoning` type)                     | `node_modules/@openrouter/sdk/esm/models/chatrequest.d.ts:50-56`                                    |
| `POST /api/v1/responses` (beta ŌĆö our target) | `effort`, `summary`, `enabled`, `maxTokens` (`ReasoningConfig`) | `node_modules/@openrouter/sdk/esm/models/reasoningconfig.d.ts:7-12` and `responsesrequest.d.ts:117` |

`callModel` targets `/responses` (spike 5.S2 confirmed via `@openrouter/sdk/esm/funcs/betaResponsesSend.js:28`). So the richer four-field shape is what we have access to.

**Docs/SDK mismatch worth noting:** the docs page documents `exclude: boolean`; the typed SDK `ReasoningConfig` (Speakeasy-generated, `reasoningconfig.js:10-19`) does **not** include `exclude`. The outbound schema validates and remaps via `remap$(v, { maxTokens: "max_tokens" })` and silently drops unknown fields. If we need `exclude` at the wire layer, we'd have to bypass the typed `callModel` and assemble the request body manually (or upstream a fix to the generator). For Card 5.4 v1, sticking to the SDK-typed `effort` + `maxTokens` is sufficient ŌĆö `exclude` is a v2 follow-on.

Because `CallModelInput` mechanically projects every field of `ResponsesRequest` (minus `stream/tools/input`), and `ResponsesRequest.reasoning?: ReasoningConfig | null | undefined` (`responsesrequest.d.ts:117`) is present on that interface, **`callModel` already accepts `reasoning` as a typed top-level argument**. No PR or upstream issue needed against the agent SDK.

## Provider mapping (since OR normalizes)

| Provider                  | OR `reasoning.effort` value                         | Underlying provider param                                                 |
| ------------------------- | --------------------------------------------------- | ------------------------------------------------------------------------- |
| OpenAI (o-series / gpt-5) | `low` / `medium` / `high` (minimal/xhigh per model) | `reasoning_effort` (chat) / `reasoning.effort` (responses)                |
| Anthropic Claude          | `low` / `medium` / `high` (also `max_tokens`)       | `thinking.budget_tokens` (1024ŌĆō128000); `effort` translated to a budget % |
| Google Gemini 3           | `xhigh` / `high` / `medium` / `low`                 | `thinkingConfig.thinkingLevel` (xhigh ŌåÆ high; others passthrough)         |
| Alibaba Qwen              | `max_tokens`                                        | `thinking_budget`                                                         |
| xAI Grok                  | `effort`                                            | Native `reasoning_effort`                                                 |

Source: [Reasoning Tokens guide](https://openrouter.ai/docs/guides/best-practices/reasoning-tokens) (observed 2026-05-24). Per the same page: when a model doesn't support a level, OR maps to the nearest supported level ŌĆö so callers can pass `medium` against an Anthropic model that only honors token-budget and OR will translate.

This is the **best case** from the hypothesis: a normalized enum, not a per-provider rabbit hole.

## Test-call evidence

### Invocation (curl, ready-to-run)

No `OPENROUTER_API_KEY` was set in the spike environment, so no real call was made. The exact invocation we would run to validate end-to-end (any reasoning-capable model; `openai/gpt-5` chosen because the `effort` knob is most visible there):

```bash
curl -sS https://openrouter.ai/api/v1/responses \
  -H "Authorization: Bearer $OPENROUTER_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "openai/gpt-5",
    "input": [{"role":"user","content":"Solve: what is 17 * 23? Show reasoning."}],
    "stream": false,
    "store": false,
    "reasoning": { "effort": "high", "summary": "auto" }
  }' | jq '{output: .output, usage: .usage, reasoning_tokens: .usage.output_tokens_details.reasoning_tokens}'
```

Equivalent via the agent SDK (typed):

```ts
import { callModel } from '@openrouter/agent';
const result = client.callModel({
  model: 'openai/gpt-5',
  input: [{ role: 'user', content: 'Solve: what is 17 * 23?' }],
  reasoning: { effort: 'high', summary: 'auto' }, // ŌåÉ already accepted by CallModelInput
});
console.log(await result.getResponse());
```

### Expected evidence

A **Yes** verdict is confirmed by any of:

1. **Wire acceptance:** 200 OK with a normal `OpenResponsesResult` payload (no 400 / "unknown field" error).
2. **Reasoning-tokens delta:** `usage.output_tokens_details.reasoning_tokens > 0` AND the same prompt with `reasoning: { effort: "low" }` (or omitted) yields a noticeably smaller reasoning-token count. This is the strongest behavioral signal that the parameter was honored, not ignored.
3. **Reasoning summary present:** when `summary: "auto"` (or `"detailed"`) is set, the response includes a `reasoning` item in `output` (typed as `OutputReasoningItem`, `reasoningitem.d.ts`).

A **No / silently-ignored** verdict would look like: 200 OK but identical `reasoning_tokens` regardless of `effort` value, and no `reasoning` output item. This would still be wire-accepted (because the field is in the typed schema and `additionalProperties: false` is not enforced) ŌĆö so the docs + typed SDK + provider-side observability are jointly the authoritative signal that `Yes` is the right call here.

### Pre-call confidence sources (why this is "Yes" without a wire call)

1. **Generated SDK type accepts the field with an outbound schema that serializes it.** `reasoningconfig.js:10-19` is the Speakeasy-generated outbound schema ŌĆö `effort` is serialized as the enum string, `maxTokens` is camelCaseŌåÆsnake_case remapped. This is the same generation pipeline that ships the rest of OR's working API surface; the field would not be in the generator's output unless OR's OpenAPI spec marks it as accepted.
2. **Field is documented as supported with examples** at <https://openrouter.ai/docs/guides/best-practices/reasoning-tokens> (observed 2026-05-24).
3. **Independent reasoning-output types exist downstream** (`reasoningitem.d.ts`, `reasoningdeltaevent.d.ts`, `reasoningdoneevent.d.ts`, `reasoningsummarytext.d.ts`, ~14 reasoning-related model files under `node_modules/@openrouter/sdk/esm/models/`) ŌĆö confirming the response side of the wire is fully modeled. A wire that emits reasoning events is, transitively, a wire that accepts the reasoning request.

If Card 5.4 lands and we later observe in production that the field is silently ignored on some provider routes, the fallback is to set `reasoning.exclude=false` explicitly and inspect `usage.output_tokens_details.reasoning_tokens` per call ŌĆö but the typed SDK + docs evidence justifies skipping the wire-validation as a blocker.

## Verdict + Path forward

### If Yes (the verdict) ŌĆö SDK shape for Card 5.4

The existing `OpenRouterAgentRunOptions.effort: string` stub becomes the wire-up target. Two small refinements + one mapping line:

```diff
 export interface OpenRouterAgentRunOptions {
   // ŌĆ”
-  /** Phase 4.8 stub ŌĆ” */
-  effort?: string;
+  /**
+   * Reasoning depth for reasoning-capable models. OR maps to each provider's
+   * native param (OpenAI reasoning_effort, Anthropic thinking budget, etc.)
+   * and substitutes the nearest supported level when a model lacks one.
+   * Ignored by non-reasoning models.
+   */
+  effort?: 'xhigh' | 'high' | 'medium' | 'low' | 'minimal' | 'none';
 }
```

Wiring at `src/agent.ts:912`:

```diff
   const result = client.callModel({
     model,
     sessionId,
     input: [{ role: 'user' as const, content: prompt }],
     instructions,
     tools: toolsForRun,
     state,
     stopWhen: [stepCountIs(maxTurns), maxCost(maxBudgetUsd)],
+    ...(effort !== undefined && { reasoning: { effort } }),
     onTurnEnd: async (_ctx, response) => { ŌĆ” },
   });
```

That's it. The `OpenEnum` typing on `ReasoningEffort` accepts our string-literal union without `as` casts. The conditional spread keeps `reasoning` absent when `effort` is unset, so we never send a `{ reasoning: { effort: undefined } }` payload that could confuse a provider.

#### Optional v2: expose `maxTokens` / `summary` / `exclude`

If we ever want richer control, expand the option to a discriminated object:

```ts
effort?:
  | 'xhigh' | 'high' | 'medium' | 'low' | 'minimal' | 'none'
  | { effort?: 'ŌĆ”'; maxTokens?: number; summary?: 'auto' | 'concise' | 'detailed' };
```

For Card 5.4 v1, the string-enum form is sufficient and matches the Claude SDK's parity target (Anthropic exposes effort as a level enum, not a token budget, in its `query()` API).
