# Phase 5.S4 — Skills / slash commands / plugins architecture

> Issue: [#106](https://github.com/Cybourgeoisie/openrouter-agent-coder/issues/106) — investigation only, no `src/` changes.

**Question**: For Cards 5.6 (slash commands), 5.7 (skills), and 5.8 (plugins), what's the smallest SDK surface that lets a host (CLI or library consumer) enumerate, materialize, and dispatch each artifact the same way Claude Code does — without dragging Claude Code's full CLI runtime into our library?

**Verdict** (per-feature):

- **Skills**: directory walker over `.claude/skills/<name>/SKILL.md` + frontmatter-driven listing in the system prompt + on-demand body load + `${VAR}` / `` !`cmd` `` / `$N` substitution at load time. Cross-vendor [`agentskills.io` spec](https://agentskills.io/specification) is the canonical frontmatter shape; Claude Code extends it; opencode implements the same pattern in 350 lines.
- **Slash commands**: structurally a degenerate skill — flat `.md` file at `.claude/commands/<name>.md` with the same frontmatter and `$ARGUMENTS` / `$N` substitution. Claude Code [explicitly merged commands into skills](https://code.claude.com/docs/en/slash-commands#:~:text=Custom%20commands%20have%20been%20merged%20into%20skills) and ships flat-file commands as a back-compat alias. We should ship the same: one loader, one substitution engine, two file shapes.
- **Plugins**: plain JSON manifest at `.claude-plugin/plugin.json` declaring optional component paths (`skills`, `commands`, `agents`, `hooks`, `mcpServers`, `lspServers`, `outputStyles`, `experimental.*`, `dependencies`, `userConfig`). Manifest is **optional** — directory-only auto-discovery is the default. Composition is pure file aggregation: a plugin is just another root the skill/command/MCP/hook loaders walk. No bespoke plugin runtime needed.

**Date**: 2026-05-24
**Spike author**: Claude (delegated by Phase 5.S4 issue #106)
**Refines**: Cards 5.6 (slash commands), 5.7 (skills), 5.8 (plugins) in [`plans/claude-sdk-parity-roadmap.md`](../claude-sdk-parity-roadmap.md).

## TL;DR

- One loader, three artifacts. Skills are the superset; commands are the flat-file shape of a skill; plugins are a manifest+directory bundle that contributes additional skills/commands/hooks/MCP/etc. to the same registries the user's own `~/.claude/` and project `.claude/` directories feed.
- The Claude SDK surface we need is **three small modules** plus a substitution helper — total ~600 LoC by the opencode reference. Most of the complexity (cache, marketplace fetch, `${CLAUDE_PLUGIN_DATA}` lifecycle, version pinning, `userConfig` keychain storage) is **out of scope for parity v1**.
- **Recommended build order: 5.7 (skills) → 5.6 (commands) → 5.8 (plugins).** Reverses the roadmap's current `5.6 → 5.7 → 5.8` ordering. Reasoning: commands are a strict subset of skills under Claude Code's current model (a `.claude/commands/foo.md` is just a flat-file skill), so building skills first and dropping in commands as a 1-file loader is cheaper than the reverse. Plugins compose both, so it remains last.
- Refined estimates: **5.6 = ~~5–8h~~ 3–5h**, **5.7 = ~~8–12h~~ 10–14h** (was undersized; the substitution layer + tool-pre-approval + lifecycle is non-trivial), **5.8 = ~~10–15h~~ 8–12h** (was oversized; once 5.6+5.7 land, plugins are mostly path aggregation + manifest validation).
- Net Phase 5 delta from this spike: **roughly flat** (~21–31h sum vs. ~23–35h pre-spike). The reshuffle is the real value, not the hour count.
- One cross-cutting design call: the substitution layer (`$ARGUMENTS`, `$N`, `${CLAUDE_PROJECT_DIR}`, `${CLAUDE_SKILL_DIR}`, `` !`shell command` ``) is shared between skills and commands. Build it once in 5.7 and reuse in 5.6/5.8 — avoid duplicating across cards.
- We have no `${CLAUDE_PLUGIN_DATA}` precedent yet (Claude Code stores it at `~/.claude/plugins/data/<id>/`). For 5.8 v1, plumb the env var through but do not implement the persistent-dependency-install lifecycle; mark it as a v2 follow-on.

## Evidence

### Skills — the cross-vendor `agentskills.io` spec

The Agent Skills standard (originated by Anthropic, now hosted at <https://agentskills.io/specification>) is what `.claude/skills/<name>/SKILL.md` files conform to. Spec frontmatter (verbatim from <https://agentskills.io/specification>, observed 2026-05-24):

| Field           | Required | Constraints                                                                                                       |
| --------------- | -------- | ----------------------------------------------------------------------------------------------------------------- |
| `name`          | Yes      | Max 64 characters. Lowercase letters, numbers, hyphens. No leading/trailing/consecutive hyphens. Must match dir.  |
| `description`   | Yes      | Max 1024 characters. Non-empty.                                                                                   |
| `license`       | No       | License name or reference to a bundled license file.                                                              |
| `compatibility` | No       | Max 500 characters. Indicates environment requirements (intended product, system packages, network access, etc.). |
| `metadata`      | No       | Arbitrary key-value map.                                                                                          |
| `allowed-tools` | No       | Space-separated string of pre-approved tools the skill may use. (Experimental.)                                   |

Directory shape:

```
skill-name/
├── SKILL.md          # Required: YAML frontmatter + markdown body
├── scripts/          # Optional: executable code
├── references/       # Optional: documentation files loaded on demand
├── assets/           # Optional: templates, resources
└── ...
```

Progressive disclosure (spec, verbatim):

1. **Metadata** (~100 tokens): `name` + `description` loaded at startup for **all** skills.
2. **Instructions** (< 5000 tokens recommended): full `SKILL.md` body loaded when the skill is activated.
3. **Resources** (as needed): files under `scripts/` / `references/` / `assets/` loaded only when referenced.

> "Keep your main `SKILL.md` under 500 lines. Move detailed reference material to separate files." — [agentskills.io/specification](https://agentskills.io/specification)

### Skills — Claude Code's extensions on top of the spec

Source: <https://code.claude.com/docs/en/slash-commands> (the page that documents skills; commands are folded in). Claude Code adds these frontmatter fields beyond the spec, observed 2026-05-24:

| Field                      | Effect                                                                                                   |
| -------------------------- | -------------------------------------------------------------------------------------------------------- |
| `when_to_use`              | Appended to `description` in the skill listing. Both share a 1,536-char cap.                             |
| `argument-hint`            | Hint shown during autocomplete (e.g., `[issue-number]`).                                                 |
| `arguments`                | Named positional arguments. With `arguments: [issue, branch]`, `$issue` → first arg, `$branch` → second. |
| `disable-model-invocation` | `true` prevents the model from auto-loading; user must invoke with `/name`.                              |
| `user-invocable`           | `false` hides from `/` menu (model-only background context).                                             |
| `model`                    | Override model for the rest of the turn.                                                                 |
| `effort`                   | Override effort level (`low` / `medium` / `high` / `xhigh` / `max`).                                     |
| `context`                  | `fork` runs the skill in a forked subagent context (no conversation history).                            |
| `agent`                    | Subagent type to use when `context: fork`.                                                               |
| `hooks`                    | Hooks scoped to this skill's lifecycle.                                                                  |
| `paths`                    | Glob patterns limiting when the skill activates.                                                         |
| `shell`                    | `bash` (default) or `powershell` for `` !`cmd` `` blocks.                                                |

Substitutions available inside the markdown body (Claude Code docs, verbatim):

| Variable               | Description                                                                                     |
| ---------------------- | ----------------------------------------------------------------------------------------------- |
| `$ARGUMENTS`           | All arguments. If absent in body, appended as `ARGUMENTS: <value>`.                             |
| `$ARGUMENTS[N]` / `$N` | 0-indexed positional argument (shell-style quoting; `"hello world"` → one arg).                 |
| `$name`                | Named positional, from the `arguments:` frontmatter list.                                       |
| `${CLAUDE_SESSION_ID}` | Current session id.                                                                             |
| `${CLAUDE_EFFORT}`     | Current effort level.                                                                           |
| `${CLAUDE_SKILL_DIR}`  | Absolute path to the directory containing this `SKILL.md`.                                      |
| `` !`<command>` ``     | Inline shell command. Output replaces the placeholder **before** the model sees the skill body. |
| ` ```! ` (fenced)      | Multi-line shell block. Same semantics.                                                         |

> Inline `` !`cmd` `` is only recognized when `!` is at start of line or after whitespace. Substitution runs once over the original file; command output is **not** re-scanned for nested placeholders.

Discovery + precedence (Claude Code docs, verbatim):

| Location   | Path                                     | Applies to                     |
| ---------- | ---------------------------------------- | ------------------------------ |
| Enterprise | (managed settings)                       | All users in your organization |
| Personal   | `~/.claude/skills/<skill-name>/SKILL.md` | All your projects              |
| Project    | `.claude/skills/<skill-name>/SKILL.md`   | This project only              |
| Plugin     | `<plugin>/skills/<skill-name>/SKILL.md`  | Where plugin is enabled        |

Precedence: enterprise > personal > project. Plugin skills are namespaced (`plugin-name:skill-name`) so they cannot collide. **Skill > command** when names collide.

Project skills walk up from cwd to repo root, plus nested `.claude/skills/` directories under subdirs of cwd (monorepo support). Live-watch picks up adds/edits/removes without restarting the session, but only for already-watched top-level dirs.

### Slash commands — the degenerate-skill verdict

The Claude Code docs are now explicit (from the slash-commands page, observed 2026-05-24):

> **Custom commands have been merged into skills.** A file at `.claude/commands/deploy.md` and a skill at `.claude/skills/deploy/SKILL.md` both create `/deploy` and work the same way. Your existing `.claude/commands/` files keep working. Skills add optional features: a directory for supporting files, frontmatter to control whether you or Claude invokes them, and the ability for Claude to load them automatically when relevant.

Concrete example from the bundled `commit-commands` plugin in Anthropic's open-source `anthropics/claude-code` repo (`plugins/commit-commands/commands/commit.md`):

```markdown
---
allowed-tools: Bash(git add:*), Bash(git status:*), Bash(git commit:*)
description: Create a git commit
---

## Context

- Current git status: !`git status`
- Current git diff (staged and unstaged changes): !`git diff HEAD`
- Current branch: !`git branch --show-current`
- Recent commits: !`git log --oneline -10`

## Your task

Based on the above changes, create a single git commit.
…
```

Same frontmatter shape as a skill, same `` !`cmd` `` substitution semantics, no directory wrapper. A command file is just a SKILL.md without a directory. The Claude Code docs reinforce: "Files in `.claude/commands/` still work and support the same [frontmatter](#frontmatter-reference). Skills are recommended since they support additional features like supporting files."

### Plugins — manifest + auto-discovery

Source: <https://code.claude.com/docs/en/plugins-reference> (observed 2026-05-24). Manifest lives at `.claude-plugin/plugin.json`. Only `name` is required; everything else is optional. The default-locations table the docs publish:

| Component         | Default Location             | Purpose                                           |
| ----------------- | ---------------------------- | ------------------------------------------------- |
| **Manifest**      | `.claude-plugin/plugin.json` | Plugin metadata and configuration (optional)      |
| **Skills**        | `skills/`                    | Skills with `<name>/SKILL.md` structure           |
| **Commands**      | `commands/`                  | Flat `.md` skills (back-compat shape)             |
| **Agents**        | `agents/`                    | Subagent Markdown files                           |
| **Output styles** | `output-styles/`             | Output style definitions                          |
| **Themes**        | `themes/`                    | Color theme definitions (experimental)            |
| **Hooks**         | `hooks/hooks.json`           | Hook configuration                                |
| **MCP servers**   | `.mcp.json`                  | MCP server definitions                            |
| **LSP servers**   | `.lsp.json`                  | Language server configurations                    |
| **Monitors**      | `monitors/monitors.json`     | Background monitor configurations (experimental)  |
| **Executables**   | `bin/`                       | Added to Bash tool's PATH while plugin is enabled |
| **Settings**      | `settings.json`              | Default plugin-level settings                     |

Concrete `plugin.json` from `anthropics/claude-code/plugins/code-review/.claude-plugin/plugin.json` (verbatim, fetched via `gh api` on 2026-05-24):

```json
{
  "name": "code-review",
  "description": "Automated code review for pull requests using multiple specialized agents with confidence-based scoring",
  "version": "1.0.0",
  "author": { "name": "Boris Cherny", "email": "boris@anthropic.com" }
}
```

Note what's **not** there: no `commands`, `agents`, `hooks`, etc. — auto-discovery handles all of it. The plugin ships its single command at `plugins/code-review/commands/code-review.md` and Claude Code finds it via the default location. **This is the rule, not the exception**: of the 13 plugins in the Anthropic-managed marketplace, the ones we inspected all relied on auto-discovery for their commands/skills/hooks.

Inline-manifest example for hook-bearing plugins (`anthropics/claude-code/plugins/security-guidance/hooks/hooks.json`):

```json
{
  "description": "Security reminder hook that warns about potential security issues when editing files",
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "python3 ${CLAUDE_PLUGIN_ROOT}/hooks/security_reminder_hook.py"
          }
        ]
      }
    ]
  }
}
```

`${CLAUDE_PLUGIN_ROOT}` is the substitution that makes plugins relocatable; it's required for any hook script, MCP server `command`, or LSP `command` that references plugin-bundled files.

Marketplace format from `anthropics/claude-code/.claude-plugin/marketplace.json` (the registry that ships with the official plugin set):

```json
{
  "$schema": "https://json.schemastore.org/claude-code-marketplace.json",
  "name": "claude-code-plugins",
  "version": "1.0.0",
  "description": "Bundled plugins for Claude Code…",
  "owner": { "name": "Anthropic", "email": "support@anthropic.com" },
  "plugins": [
    {
      "name": "agent-sdk-dev",
      "description": "Development kit for working with the Claude Agent SDK",
      "source": "./plugins/agent-sdk-dev",
      "category": "development"
    }
    /* … 12 more entries, each with name + source + category … */
  ]
}
```

So a marketplace is just `{ name, plugins[] }` where each entry points at a source (relative path, git ref, npm package, or URL). The plugin store dereferences sources to the local cache (`~/.claude/plugins/cache/<plugin>/<version>/`) on install. For v1 parity, **we don't need to implement the marketplace fetcher** — direct path-based plugin loading (Claude's `--plugin-dir <path>`) covers the use case; marketplaces are a v2.

### opencode reference — confirms the converged-loader approach

opencode's `packages/opencode/src/skill/index.ts` (353 lines, fetched 2026-05-24) walks both `.claude/skills/**/SKILL.md` (for Claude-format skills) and a parallel `.agents/skills/**/SKILL.md` (their own namespace), starting from `$HOME` and walking down the project tree. Discovery globs (`oc-skill-index.ts:21-25`):

```ts
const CLAUDE_EXTERNAL_DIR = '.claude';
const AGENTS_EXTERNAL_DIR = '.agents';
const EXTERNAL_SKILL_PATTERN = 'skills/**/SKILL.md';
const OPENCODE_SKILL_PATTERN = '{skill,skills}/**/SKILL.md';
const SKILL_PATTERN = '**/SKILL.md';
```

Frontmatter is parsed with their `ConfigMarkdown.parse` (their gray-matter equivalent). The validator (lines 52-58) is intentionally permissive — only requires `name: string` and accepts `description?: string`:

```ts
function isSkillFrontmatter(data: unknown): data is { name: string; description?: string } {
  return (
    isRecord(data) &&
    typeof data.name === 'string' &&
    (data.description === undefined || typeof data.description === 'string')
  );
}
```

The `Skill.fmt(list, { verbose })` helper (lines 326-351) is how skills surface to the model — formatted as a "## Available Skills" markdown block (or `<available_skills>` XML in verbose mode) injected into the system prompt. **This is the entire surfacing mechanism** — no tool calls, no on-demand fetch protocol, just system-prompt injection of the listing.

opencode also has a `Discovery.pull(url)` that downloads remote skill bundles via an `index.json` manifest (`oc-skill-discovery.ts:54-104`) — that's their equivalent of plugin marketplaces, scoped tighter (skills-only, no full plugin packaging).

opencode's command system unifies skills + commands + MCP prompts into one menu (`packages/opencode/src/command/index.ts:14-39`):

```ts
export const Info = Schema.Struct({
  name: Schema.String,
  description: Schema.optional(Schema.String),
  agent: Schema.optional(Schema.String),
  model: Schema.optional(Schema.String),
  source: Schema.optional(Schema.Literals(['command', 'mcp', 'skill'])),
  template: Schema.Unknown, // lazy promise — MCP prompts resolve on demand
  subtask: Schema.optional(Schema.Boolean),
  hints: Schema.Array(Schema.String),
}).annotate({ identifier: 'Command' });
```

And the loader's last step (lines 141-152) iterates skills and registers them as commands if no command of the same name already exists:

```ts
for (const item of yield * skill.all()) {
  if (commands[item.name]) continue;
  commands[item.name] = {
    name: item.name,
    description: item.description,
    source: 'skill',
    get template() {
      return item.content;
    },
    hints: [],
  };
}
```

**This is the converged model.** A user's "what slash commands are available" view contains: built-in commands (init, review), config-file commands, MCP prompts, and every skill on disk. Our 5.6 / 5.7 implementations should converge the same way.

opencode does **not** implement Claude's plugin manifest (`.claude-plugin/plugin.json`) at all. Their `packages/opencode/src/plugin/` directory contains AI provider adapters (codex, azure, cloudflare, …), not Claude-Code-style plugins. So opencode is a useful reference for skills + commands but **not** for plugins; for plugins we rely on Claude Code's reference + the spec docs.

### codex reference — not relevant

`openai/codex` ships in Rust (`codex-rs/`) and has no equivalent of skills / slash commands / plugins as user-extensible primitives — the CLI's special commands (`/init`, `/review`, …) are hard-coded handlers in their TUI. No cross-vendor convergence to draw on. Same finding as 5.2.S1.

## SDK-surface sketches

These are the minimum public surfaces our library should expose for each card. Designs target a host CLI that wants to wire up skills/commands/plugins without inheriting Claude Code's full runtime — the same shape our existing `OpenRouterAgent` consumer pattern uses.

### 5.7 — Skills (build first)

```ts
// src/skills/spec.ts
export interface SkillFrontmatter {
  name: string; // ^[a-z0-9](?:[a-z0-9-]{0,62}[a-z0-9])?$
  description?: string;
  whenToUse?: string; // `when_to_use` — appended to description in listing
  license?: string;
  compatibility?: string;
  metadata?: Record<string, string>;
  allowedTools?: string; // space-separated; same syntax as Card 3.2
  argumentHint?: string;
  arguments?: string[]; // named positional, accepts space-separated or YAML list
  disableModelInvocation?: boolean;
  userInvocable?: boolean; // default true
  model?: string; // overrides session model
  effort?: EffortLevel; // reuses Card 5.4 enum
  context?: 'fork';
  agent?: string; // subagent type when context: fork
  paths?: string[]; // glob filters for auto-activation
  shell?: 'bash' | 'powershell';
}

export interface SkillInfo {
  name: string; // namespaced as `pluginName:name` when sourced from a plugin
  source: 'user' | 'project' | 'plugin' | 'enterprise' | 'builtin';
  pluginName?: string;
  location: string; // absolute path to SKILL.md
  frontmatter: SkillFrontmatter;
  body: string; // raw markdown body (substitutions not yet applied)
}

// src/skills/loader.ts
export interface SkillLoaderOptions {
  cwd: string; // walk up from here looking for .claude/skills/
  home?: string; // defaults to os.homedir()
  pluginRoots?: Array<{ name: string; root: string }>; // populated by 5.8
  disableUserSkills?: boolean;
  disableProjectSkills?: boolean;
}

export interface SkillLoader {
  list(): Promise<SkillInfo[]>; // metadata only — what goes in the listing
  get(name: string): Promise<SkillInfo | undefined>; // body included
  render(name: string, args: string[], substitutions: SubstitutionContext): Promise<string>; // executes !`cmd` blocks + variable substitution
  watch(onChange: (event: SkillChangeEvent) => void): () => void; // live-reload (no-op for v1)
}

export function createSkillLoader(opts: SkillLoaderOptions): SkillLoader;
```

Where it plugs in: `OpenRouterAgentRunOptions` gains an optional `skills?: SkillLoader` (or `skillsDir?: string` convenience). The agent's instruction-builder injects the listing block (see "Trigger / loading" below); a new `Skill` client tool calls `loader.render()` on invocation and feeds the rendered body back into the conversation. Mirrors Card 3.4's CLAUDE.md auto-discovery — same `cwd`-up-walk pattern, same enterprise/personal/project precedence.

### 5.6 — Slash commands (build second)

A command is a flat-file skill — same loader, narrower discovery glob:

```ts
// src/commands/loader.ts
export interface CommandLoaderOptions {
  cwd: string;
  home?: string;
  pluginRoots?: Array<{ name: string; root: string }>;
  skillLoader?: SkillLoader; // converged-menu: skill-as-command (opencode pattern)
}

export interface CommandInfo {
  name: string;
  description?: string;
  argumentHint?: string;
  source: 'user' | 'project' | 'plugin' | 'skill' | 'builtin';
  // ... reuses skill shape, no body distinction needed
}

export interface CommandLoader {
  list(): Promise<CommandInfo[]>;
  resolve(input: string): Promise<{ name: string; args: string[]; body: string } | undefined>;
  // input is the raw line after the leading "/" — splits name + args (shell-quoted), renders body
}
```

`resolve()` is what the host CLI calls when a user types `/foo bar "baz qux"`: it returns the rendered prompt body that should become the next `prompt` argument to `OpenRouterAgent.run({ prompt })`. Commands are **host-invoked only** — the model doesn't trigger them (skills do auto-trigger; that's the difference per Claude Code's design).

### 5.8 — Plugins (build last)

```ts
// src/plugins/spec.ts
export interface PluginManifest {
  name: string; // required; everything else optional
  displayName?: string;
  version?: string;
  description?: string;
  author?: { name?: string; email?: string; url?: string };
  homepage?: string;
  repository?: string;
  license?: string;
  keywords?: string[];

  // Component overrides (all optional — auto-discovery is the default)
  skills?: string | string[]; // adds to default skills/
  commands?: string | string[]; // replaces default commands/
  agents?: string | string[]; // replaces default agents/
  hooks?: string | object; // replaces default hooks/hooks.json
  mcpServers?: string | object; // replaces default .mcp.json
  outputStyles?: string | string[];
  lspServers?: string | object;

  // v2 (declared but not implemented in 5.8 v1)
  userConfig?: Record<string, UserConfigOption>;
  dependencies?: Array<string | { name: string; version?: string }>;
  experimental?: { themes?: string | string[]; monitors?: string | string[] };
}

// src/plugins/loader.ts
export interface PluginLoaderOptions {
  pluginDirs: string[]; // explicit paths (`--plugin-dir` equivalent)
  marketplaceFile?: string; // path to marketplace.json (v2; ignored in v1)
}

export interface LoadedPlugin {
  manifest: PluginManifest;
  root: string; // absolute path → ${CLAUDE_PLUGIN_ROOT}
  dataDir: string; // absolute path → ${CLAUDE_PLUGIN_DATA} (declared, not auto-managed in v1)

  // Aggregated contributions, ready to fold into the other loaders:
  skillRoots: string[];
  commandRoots: string[];
  agentRoots: string[];
  hookConfigs: HookConfig[]; // inline + file-resolved
  mcpServers: Record<string, McpServerSpec>; // 5.2.3-shaped
}

export function loadPlugins(opts: PluginLoaderOptions): Promise<LoadedPlugin[]>;
```

`OpenRouterAgentRunOptions` gains `plugins?: LoadedPlugin[]`; the agent constructor folds plugin contributions into the skill / command / hook / MCP registries it already builds. No bespoke "plugin runtime" — plugins are just composition. The model never knows the difference between a user-installed skill and a plugin-shipped skill except via the namespace prefix (`pluginName:skillName`).

### Substitution helper (shared)

```ts
// src/substitution/render.ts
export interface SubstitutionContext {
  arguments: string[]; // already shell-split
  named?: Record<string, string>; // from `arguments:` frontmatter
  sessionId: string;
  effort?: EffortLevel;
  skillDir?: string; // absolute path to SKILL.md's dir
  pluginRoot?: string; // absolute path to plugin root
  pluginData?: string; // ~/.claude/plugins/data/<id>/
  projectDir: string; // cwd / repo root
  userConfig?: Record<string, string | number | boolean>;
  env?: Record<string, string>; // process.env passthrough
  shell?: 'bash' | 'powershell';
  signal: AbortSignal;
  timeoutMs?: number; // per-shell-block
}

export function renderSkillBody(body: string, ctx: SubstitutionContext): Promise<string>;
```

`renderSkillBody` applies, in order:

1. `${VAR}` interpolation for the documented variables (`CLAUDE_SESSION_ID`, `CLAUDE_PROJECT_DIR`, `CLAUDE_SKILL_DIR`, `CLAUDE_EFFORT`, `CLAUDE_PLUGIN_ROOT`, `CLAUDE_PLUGIN_DATA`, `user_config.<key>`, generic `${ENV_VAR}`).
2. `$ARGUMENTS` and `$ARGUMENTS[N]` / `$N` / `$name` positional substitution.
3. `` !`<cmd>` `` inline shell evaluation, executed at the same trust level as our existing `run_command` tool. Position-restricted (start of line or after whitespace; not after non-whitespace chars). Output is **not** re-scanned. (One pass; matches Claude Code's documented behavior.)
4. Fenced ` ```! ` blocks (multi-line shell).

The shell-execution step is the one with material risk — it should:

- Honor a `disableSkillShellExecution: true` settings flag (Claude Code parity — replaces the command with `[shell command execution disabled by policy]`).
- Run under the host's existing permission machinery (Cards 3.1–3.2) — a skill's `allowed-tools: Bash(git:*)` grants pre-approval for the matching command lines only.
- Respect `AbortSignal` from the agent run; default per-block timeout (60s) configurable.

## Trigger / loading / composition design

### Skills — how the model invokes one

Two paths the model can take, mirroring Claude Code:

1. **Auto-invocation**: every loaded skill's `(name, description, when_to_use)` triple is materialized into a `## Available Skills` block (or `<available_skills>` XML) and concatenated to the existing instructions. The model decides whether to "load" a skill — implementation: a new built-in client tool `load_skill(name: string, args?: string)` that returns the rendered body. The model calls this tool, receives the body as the tool result, and proceeds.
2. **User-invocation**: when the host CLI sees `/<name>` it calls `loader.render(name, args, ctx)` and passes the result as the next `prompt`. No model loop spin-up to load — the body is already inlined.

opencode does (1) inline (full SKILL.md body in the listing block from the start) for small skill counts and only falls back to on-demand for very large skill sets. Claude Code does the opposite: only `(name, description)` in the listing; body loads on demand via the `Skill` tool. **Recommendation: Claude Code's approach.** Reasons:

- Aligns with the agentskills.io progressive-disclosure design (~100 tok metadata per skill, ~5k tok body when invoked).
- Avoids blowing up the system prompt on projects with many skills.
- Matches what Card 5.1 (compaction) expects — skills can be re-attached post-compact within a token budget.

Implementation: ship a `Skill` tool (singleton; description listing comes from `SkillLoader.list()`). Add a `skillDescriptionBudget` option mirroring Claude Code's `skillListingBudgetFraction` (default 1% of the model's context window). Skills whose listing was dropped are still callable by exact name but won't auto-trigger.

### Skills — `context: fork` and subagent integration

Claude Code's `context: fork` runs a skill in a forked subagent — Card 4.7 (subagent system) gives us the primitive. The skill body becomes the subagent's `prompt`; `agent` field selects which subagent type (`Explore`, `Plan`, `general-purpose`, or a custom one from `.claude/agents/`). Cards 4.7–4.9 already shipped, so 5.7 can wire `context: fork` directly into the existing subagent code path. **No new subagent infrastructure needed.**

### Commands — how the host CLI invokes one

Pure host responsibility. The library's `CommandLoader.resolve('/foo bar')` returns `{ name, args, body }`; the host:

1. Echoes the resolved name to the user (autocomplete UX).
2. Feeds `body` as the next `prompt` to `OpenRouterAgent.run({...})`.

Model never sees the command machinery — commands are pre-resolved into prompts before the model loop starts. (Skills are different: the model can call the `Skill` tool mid-loop.)

### Plugins — composition rules

Per the Claude Code path-behavior table (verbatim from the plugins-reference docs):

- **Replaces the default**: `commands`, `agents`, `outputStyles`, `experimental.themes`, `experimental.monitors`. If the manifest specifies `commands: ["./extras/"]`, the default `commands/` is **not scanned**. Keep both: `commands: ["./commands/", "./extras/"]`.
- **Adds to the default**: `skills`. Default `skills/` is always scanned; manifest paths layer on.
- **Own merge rules**: `hooks`, `mcpServers`, `lspServers` (each section documents its own merge behavior).

For v1, implement these rules faithfully — they're cheap (path-array union vs. replacement) but observable (a plugin author following Claude Code's docs will expect these semantics).

Auto-discovery rule (Claude Code v2.1.142+): a plugin root with `SKILL.md` directly inside (no `skills/` subdir, no `skills` manifest key) is loaded as a single-skill plugin. Worth supporting in v1; tiny code.

### `${CLAUDE_PLUGIN_DATA}` lifecycle — defer to v2

The persistent data directory at `~/.claude/plugins/data/<id>/` survives plugin updates and is the recommended place for `node_modules` / Python venvs / caches. The recommended `SessionStart` hook pattern (per the docs) installs deps on first run + on manifest changes via a `diff` check.

For 5.8 v1: declare the env var, resolve it to the deterministic path, but do not implement the auto-cleanup-on-uninstall lifecycle or version-cache management. Mark as a v2 follow-on. Most plugins don't use `${CLAUDE_PLUGIN_DATA}`; the ones that do work without our cleanup, just leak a directory if the user uninstalls.

### `userConfig` + `channels` + `dependencies` — defer to v2

These three plugin-manifest features carry independent implementation cost:

- `userConfig` requires a prompt-on-enable UI flow (host-CLI concern; library can only define the schema), keychain storage for `sensitive: true` values, and substitution into MCP/hook commands as `${user_config.KEY}`.
- `channels` binds an MCP server to a message-injection channel — depends on the agent loop's ability to receive out-of-band user messages, which is itself Card 5.3 (streaming input) territory.
- `dependencies` is a plugin-dependency resolver. Useful for ecosystems with many plugins; overkill for v1.

For 5.8 v1: parse + validate the schema for all three (so manifests don't error), but don't implement runtime behavior. Mark as v2.

## Cross-card design decisions

1. **Build order 5.7 → 5.6 → 5.8.** Reverses roadmap order. Skills are the superset; commands are degenerate skills; plugins compose both. **Sequence rule for the roadmap: 5.6 depends on 5.7; 5.8 depends on both.**
2. **Single substitution helper.** Built in 5.7, reused by 5.6 + 5.8. Don't duplicate `$ARGUMENTS` / `${CLAUDE_*}` / `` !`cmd` `` parsing across cards. The helper lives at `src/substitution/` and is exported only as an internal module.
3. **Namespace plugin-sourced skills as `pluginName:skillName`.** Avoids collision with user/project skills. Mirrors Claude Code and matches our existing tool-naming convention (`mcp__server__tool`). The `name` field in the SKILL.md frontmatter stays the unqualified name; the loader prepends the namespace when sourcing through 5.8.
4. **Listing-only by default in the system prompt.** `(name, description, when_to_use)` triple per skill, capped at 1,536 chars per Claude Code's spec, total budget at 1% of model context. The full SKILL.md body loads via a `Skill` built-in tool. Don't full-inline by default — kills context budget on large skill sets.
5. **Commands are pre-resolved by the host.** The model doesn't see commands as a separate primitive; the `/<name>` input is resolved into a `prompt` argument before the agent run starts. Skills, by contrast, **are** model-callable mid-loop (via the `Skill` tool). This dichotomy matches Claude Code and is a deliberate UX call: commands are user shortcuts; skills are model capabilities.
6. **Frontmatter parser is shared.** Used by skills (5.7), commands (5.6 — same shape, fewer fields used), agents (already shipped Card 4.7 — `.claude/agents/<name>.md`), and plugin sub-components. Pick one approach: hand-rolled (30 lines, no deps) or `gray-matter` (~250 KB, dep-heavy but battle-tested). **Recommendation: hand-rolled.** Frontmatter is line-delimited `---` blocks with YAML inside; we already have Zod for validation, so the "parser" is `split('---', 3)` + `YAML.parse`. The `yaml` package (~150 KB, MIT, peer-dep-free) is the only addition.
7. **`disable-model-invocation` honors via tool description.** The `Skill` tool's listing excludes skills with `disable-model-invocation: true`. The skill is still in the registry (host can invoke it via command path), just not advertised to the model.
8. **`allowed-tools` integrates with Cards 3.1–3.2.** A skill's `allowed-tools: Bash(git:*)` is a per-skill permission grant active for the duration of the skill's invocation. Implementation: push a permission-rule scope onto the existing permission stack at skill activation; pop on completion. Mirrors how Claude Code documents the field. Plan mode (3.3) still wins — `allowed-tools` cannot grant write access in plan mode.
9. **Plugin discovery is path-based, not registry-based.** The `pluginDirs: string[]` option takes explicit paths; no marketplace.json fetcher in v1. Host CLIs that want marketplace UX wire their own; the library validates and loads from disk.
10. **`SKILL.md` files contain no executable code by themselves.** The shell-execution surface is `` !`cmd` `` and `scripts/` references (which the model triggers via tool calls). Both run under our existing permission + abort-signal infrastructure. Auditing risk surface: skill frontmatter shouldn't grant escalations the user hasn't already given the agent. The `allowed-tools` pre-approval is the one new authorization surface; it's bounded by the user's overarching `allowedTools` config (Card 3.2) — a skill cannot grant access to a tool the user has disallowed.

## Risks + caveats

1. **Spec drift between [agentskills.io](https://agentskills.io) and Claude Code's extensions.** The base spec is stable but Claude Code keeps adding fields (`paths`, `shell`, `hooks`, `effort`, `context`). Strategy: accept-but-warn for unknown frontmatter fields (mirrors Claude Code's "warnings, not errors" plugin-manifest behavior); keep the recognized-fields table colocated with the parser so adding a field is one PR.
2. **Shell execution risk surface.** `` !`cmd` `` runs at the same trust level as our existing tools but with substitution. Care needed: the user-supplied `$N` arguments must **not** be interpolated into the command line as shell-substituted strings (command injection). Substitution happens in the **markdown body**, not in the shell command itself — that is, `` !`echo $0` `` would expand `$0` as a shell variable (empty), not as our `$ARGUMENTS[0]`. This matches Claude Code; document it clearly so skill authors don't expect arg-into-shell substitution.
3. **Live-watch is fragile.** Claude Code only watches top-level skill dirs that existed at session start. We can match that exactly (cheap), but the resulting UX has gotchas. Document the "create top-level dir → restart" requirement; ship a `forceReload()` API on the loader for hosts that want explicit control.
4. **Conflict with `--add-dir`.** Claude Code treats `--add-dir` directories as file-access grants only, but makes a documented exception for `.claude/skills/` (which **are** discovered). For parity, our loader should treat host-supplied "additional dirs" the same way. Not in scope for 5.7 itself (we don't have `--add-dir` yet), but document the convention so a future "extra-dirs" feature lands cleanly.
5. **Plugin manifest version drift.** Manifest schema may add fields between releases. Strategy: ignore unknown top-level fields (don't fail load), warn in validate (matches Claude Code's `claude plugin validate` behavior), expose a `--strict` mode that escalates warnings to errors. One-time implementation cost; saves future-us from being a churn-blocker.
6. **`${CLAUDE_PLUGIN_DATA}` not auto-managed.** If a plugin uses it and the user uninstalls, the data directory leaks until manual cleanup. Risk is small (disk-space leak, not security); acceptable for v1.
7. **Frontmatter YAML edge cases.** `arguments: [a, b]` vs. `arguments: "a b"` (both valid per Claude Code docs); `allowed-tools` accepts both string and array shapes; `paths` accepts comma-separated string or YAML list. The parser must accept both shapes for each field — already implied above but cheap to get wrong. Cover with unit tests.
8. **Convergence with Card 5.1 compaction.** Claude Code's compaction re-attaches the most recent invocation of each skill within a 25k-tok budget. Our 5.7 implementation should expose the rendered-body cache via a stable identifier the compaction layer (Card 5.1) can read. Coordinate with 5.1 before shipping 5.7 — minor coupling, but worth a 30-min design sync.

**Build all three. Reorder to 5.7 → 5.6 → 5.8. Refined sizes: 10–14h / 3–5h / 8–12h respectively.**
