# Phase 5.2.S1 — MCP client vendor decision (`@modelcontextprotocol/sdk` vs. roll-our-own)

> Issue: [#98](https://github.com/Cybourgeoisie/openrouter-agent-coder/issues/98) — investigation only, no `src/` changes.

**Question**: For Cards 5.2.1 (stdio transport) and 5.2.2 (HTTP+SSE transport), should we depend on `@modelcontextprotocol/sdk` (Anthropic's official MCP client/server SDK) or implement transports + JSON-RPC framing + lifecycle ourselves?

**Verdict**: **Vendor.** Use `@modelcontextprotocol/sdk` v1.29.0 with **client-only subpath imports** (`@modelcontextprotocol/sdk/client/{index,stdio,sse,streamableHttp}.js`). The SDK is published by Anthropic, MIT-licensed, ships all three transports we need (stdio + deprecated-but-still-needed SSE + Streamable HTTP), and is already the de facto choice across the ecosystem (opencode imports it directly, Anthropic's own `anthropic-sdk-typescript` and `claude-code-action` consume it). The install footprint is non-trivial (~21 MB of transitive deps the client subtree never imports at runtime — `express`, `hono`, `jose`, `cors`) but is bounded to the install graph; bundle-size doesn't apply since this is a CLI, not a browser app. Rolling our own would re-implement ~2,600 lines of JSON-RPC framing, capability negotiation, progress tokens, reconnection, and OAuth across three transports — far more than the listed 16h for 5.2.1+5.2.2 combined.

**Date**: 2026-05-24
**Spike author**: Claude (delegated by Phase 5.2.S1 issue #98)
**Refines**: Cards 5.2.1 (MCP transport: stdio) and 5.2.2 (MCP transport: HTTP+SSE).

## TL;DR

- **Vendor** `@modelcontextprotocol/sdk@^1.29.0` (MIT, by Anthropic PBC).
- Import only from `client/*` subpath exports — the heavy server-side deps (express, hono, jose, cors) are not loaded at runtime when only client modules are imported, though they remain on disk.
- Peer-dep `zod ^3.25 || ^4.0` is satisfied by our existing `zod ^4.0.0` ([`package.json:38`](../../package.json)).
- Refined estimates: **5.2.1 = 3–4h** (was 8h), **5.2.2 = 4–5h** (was 8h). Net savings ~7–9h across the two cards.
- Recommended architecture: a thin `src/mcp/transport.ts` factory returning an SDK `Client` instance bound to a transport; tool-bridge (5.2.4) consumes `client.listTools()` + `client.callTool()`; lifecycle hooks (5.2.5) wrap `connect()` / `close()`.

### Transport class shape (what the adapter layer would wrap)

`stdio` transport (`client/stdio.d.ts:46-76`):

```ts
export declare class StdioClientTransport implements Transport {
  constructor(server: StdioServerParameters);
  onclose?: () => void;
  onerror?: (error: Error) => void;
  onmessage?: (message: JSONRPCMessage) => void;
  start(): Promise<void>;
  get stderr(): Stream | null;
  get pid(): number | null;
  close(): Promise<void>;
  send(message: JSONRPCMessage): Promise<void>;
}
```

`StreamableHTTPClientTransport` (`client/streamableHttp.d.ts:97-170`) implements the same `Transport` interface but with `resumptionToken` / `Last-Event-ID` replay, reconnection backoff, `Mcp-Session-Id` header handling, and optional OAuth. `SSEClientTransport` is the deprecated POST-+-EventSource pair (`client/sse.d.ts:52-80`).

Stdio framing is line-delimited JSON (`shared/stdio.js:25-30` — `serializeMessage = json + '\n'`, `deserializeMessage = JSONRPCMessageSchema.parse(JSON.parse(line))`). That single file is genuinely small (~30 lines) — but it's the only small piece.

The heavy deps live exclusively under `dist/esm/server/` (express, hono, cors, jose) and are not pulled into the client module graph at runtime. So **bundle/runtime cost for our use is small** — just the SDK package itself (6 MB) + `eventsource` (188 KB) + `cross-spawn` (small) + `ajv` (2.5 MB, used for tool I/O validation by the `Client` constructor's default `jsonSchemaValidator`).

### Install-graph cost

The heavy server deps **are** in `dependencies` (not `optionalDependencies` or `peerDependenciesMeta.optional`), so they install regardless of which subpath we import. That adds ~21 MB to `node_modules` on disk and pulls in ~75 packages we'll never execute. For a CLI tool (this repo's target), that's a one-time install hit, not a per-invocation cost. Acceptable trade-off; not enough to justify rolling our own.

### Bundle size when shipped to consumers

Our `dist/` is the consumer-facing build. If a downstream library consumer doesn't use MCP features, they don't `import` from `src/mcp/*`, so tree-shaking + dynamic-import gating keeps the SDK out of their bundle. If they do, they pull in the SDK as a transitive dep — same cost they'd pay anywhere else. We can also lazy-load the SDK behind a dynamic `import('@modelcontextprotocol/sdk/client/index.js')` in the MCP entry point to keep the cold-start path zero-cost for users without configured MCP servers. (Recommended pattern; ~0.5h to wire in 5.2.1.)

### Card 5.2.1 cost breakdown (refined)

- Add `@modelcontextprotocol/sdk` as a runtime dep + lockfile bump: **0.25h**
- `src/mcp/transport-stdio.ts` factory: `(params: StdioServerParameters) => Client` returning a connected `Client` instance (`new Client({…}) ; await client.connect(new StdioClientTransport(params))`): **0.5h**
- Subprocess lifecycle wiring — surface `stderr` via our existing logger (the SDK exposes a `stderr` PassThrough on the transport instance — `client/stdio.d.ts:66`), propagate close errors through our error pipeline: **1h**
- Tests: spin up the `@modelcontextprotocol/sdk` example "everything server" via stdio in a child process (the SDK ships one under `dist/esm/examples/`), assert `listTools()` / `callTool()` round-trips, assert clean process shutdown on `close()`: **1.5–2h**
- README + parity matrix note: **0.25h**

### Card 5.2.2 cost breakdown (refined)

- `src/mcp/transport-http.ts` factory wrapping `StreamableHTTPClientTransport` + the deprecated `SSEClientTransport` fallback (since some MCP servers still only speak SSE): **1h**
- Pass-through of `requestInit` (headers, custom fetch) so users can BYO auth: **0.5h**
- Reconnection-options passthrough: **0.25h**
- Tests: mock HTTP server (use Node's `http.createServer`) that speaks the Streamable HTTP framing, assert reconnect-with-`Last-Event-ID` works; second test for the SSE fallback: **2–3h**
- README + parity matrix note: **0.25h**

### Why not closer to "1h each"

The transports themselves are one-liners over the SDK constructors, but **testing** transport adapters honestly requires standing up a real MCP-speaking server in the test harness (mocking the wire protocol defeats the purpose of vendoring a well-tested SDK). The SDK's bundled `inMemory` transport (`dist/esm/inMemory.{d,j}s`) gives us a free in-memory loopback for the bridge-layer tests (5.2.4), but Cards 5.2.1 and 5.2.2 specifically test wire-level fidelity, so they each need a real process / real socket. That's where the 3–5h floor comes from.

### Implications for 5.2.3 / 5.2.4 / 5.2.5

The vendor decision doesn't change the listed estimates for the downstream cards much, but it does **reduce risk**:

- **5.2.3 (`.mcp.json` discovery)** — schema is just `Record<string, StdioServerParameters | { url: string }>`, validated via Zod against the SDK's typed shape. Stays at 5h, maybe shrinks to 4h.
- **5.2.4 (tool-bridge)** — translates `ListToolsResultSchema` outputs to our `Tool[]`. The SDK gives us a JSON-Schema `inputSchema` per tool; we already have a `jsonSchema → Zod` bridge from Card 3.5. Stays at 8h.
- **5.2.5 (lifecycle hooks)** — `McpServerStart` / `McpServerStop` hooks wrap `client.connect()` and `client.close()`. Stays at 6h.

Refresh in roadmap: only 5.2.1 + 5.2.2 lines need numeric updates; 5.2.3–5.2.5 stay put but get a "(vendor: @modelcontextprotocol/sdk per [5.2.S1](./spikes/5.2.S1-mcp-vendor.md))" note in their description.

## Recommended architecture sketch

Module layout under `src/mcp/`:

```
src/mcp/
  index.ts            # public exports (McpClient, connectMcpServer, McpServerSpec)
  spec.ts             # Zod schema for .mcp.json entries (stdio | http variant)
  transport-stdio.ts  # factory → connected Client over StdioClientTransport
  transport-http.ts   # factory → connected Client over StreamableHTTP (+ SSE fallback)
  config.ts           # .mcp.json discovery + load + validate (Card 5.2.3)
  bridge.ts           # MCP Tool → our Tool[] mapping (Card 5.2.4)
  hooks.ts            # McpServerStart/Stop hook definitions (Card 5.2.5)
  *.test.ts           # one per module
```

### Key boundary decisions

1. **Lazy-load the SDK.** All imports of `@modelcontextprotocol/sdk` live behind a dynamic `import()` inside `connectMcpServer()`. Cold-start for users without `.mcp.json` is zero extra cost. Add `@modelcontextprotocol/sdk` to `optionalDependencies` if we want pure zero-cost — but plain `dependencies` is simpler and the SDK install is one-time-per-user.

2. **`McpClient` is a thin wrapper, not a re-export.** Expose a narrow surface (`listTools()`, `callTool()`, `close()`) that hides the SDK's `Protocol` machinery. This keeps the option of swapping the SDK out later if (hypothetically) it goes unmaintained.

3. **`StateAccessor` integration** — MCP server connections are out-of-band state, not turn state. They live on the `AgentRunner` instance, not on `ConversationState`. Lifecycle is: start servers at run start (or first tool call to that server), shut them down on run end. Cards 5.2.5's hooks expose that boundary.

4. **Tool-bridge contract** — Card 5.2.4 should translate each MCP tool into our `Tool` shape **eagerly at server-connect time**, not lazily. The SDK's `client.listTools()` result is cached on the `Client` instance; we mirror that into our tool registry once and refresh only on `tools/list_changed` notifications (the SDK exposes this via `ListChangedHandlers` in `ClientOptions` — `client/index.d.ts:83`).

5. **OAuth (auth.ts)** — deferred. Not needed for v1 (`.mcp.json` typically points at local stdio servers or self-hosted HTTP). Mark as a follow-on; the SDK already provides `OAuthClientProvider` we can wire in later without breaking the v1 surface.

### Failure surface

The SDK throws typed errors (`StreamableHTTPError`, `SseError`, `UnauthorizedError`). Our wrapper should:

- Catch these at the `connect()` boundary, surface as our own `McpConnectionError` with the server name attached.
- Catch `callTool()` errors and re-throw with the originating tool name + server name (the SDK's errors are about the wire layer, not "which tool you were running").
- Treat transport `onerror` callbacks as the server crashing; emit a `McpServerStop` hook event and remove tools from the registry. (Card 5.2.5.)

## Risks + caveats

1. **SDK version churn.** `@modelcontextprotocol/sdk` ships fast (1.29 today, was 1.27 when opencode pinned). API has been stable on the `Client` surface; transport classes added Streamable HTTP at v1.10ish and SSE was deprecated. We should pin a minor and update deliberately; the surface we depend on is small enough that breakage is detectable in CI.

2. **Server-deps in install graph.** 21 MB of unused express/hono/jose on disk is a smell but not a blocker. If it ever matters, we can vendor only the four client files into our tree and break the SDK dep — that's a "1 day refactor when it becomes painful" escape hatch, not a today-decision.

3. **Spec drift.** The MCP spec is still pre-1.0. The SDK tracks the spec; we'd otherwise have to track it ourselves. This is the strongest "vendor" argument: protocol-spec maintenance is a perpetual cost we don't have the bandwidth for.

4. **Bundle size for library consumers.** If someone uses this repo as a library and doesn't want MCP, they shouldn't pay for it. Lazy-loading the SDK behind a dynamic `import()` (recommended above) handles this — tree-shakers will drop the SDK from the bundle entirely if no MCP entry point is imported.

## Verdict + Path forward

**Vendor `@modelcontextprotocol/sdk@^1.29.0`.**

Concrete actions for Cards 5.2.1 + 5.2.2:

1. `npm install @modelcontextprotocol/sdk@^1.29.0` (5.2.1 adds it; 5.2.2 reuses).
2. Implement `src/mcp/transport-stdio.ts` and `src/mcp/transport-http.ts` as thin factories returning connected `Client` instances.
3. Lazy-load the SDK behind a dynamic `import()` so cold-start cost is zero for users without `.mcp.json`.
4. Test against the SDK's bundled example server for end-to-end fidelity; use the `inMemory` transport for fast bridge-layer tests in 5.2.4.
5. Pin the SDK to a known-good minor and update on a deliberate cadence.

Update the roadmap (`plans/claude-sdk-parity-roadmap.md`) Cards 5.2.1 / 5.2.2 estimate lines from `8h` to `3–4h` and `4–5h` respectively, with a backlink to this spike.
