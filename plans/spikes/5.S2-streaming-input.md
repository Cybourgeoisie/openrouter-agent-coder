# Phase 5.S2 — Streaming-input feasibility spike

> Issue: [#63](https://github.com/Cybourgeoisie/openrouter-agent-coder/issues/63) — investigation only, no `src/` changes.

**Question**: Does `@openrouter/agent` `callModel` (or any related SDK surface) accept mid-call user-message injection — the streaming-input pattern where the host pushes additional user messages into an already-running run?

**Verdict**: **No (workaround exists).** `callModel` takes a single `input` per invocation; there is no `push` / `send` / `input(...)` method on `ModelResult`. The SDK *does* expose an `interruptedBy` state flag that lets an external process cleanly stop a run mid-stream, save partial state, and resume via a fresh `callModel` call. That gives us an "interrupt-then-restart-with-appended-prompt" path that we can hide under a Claude-SDK-shaped facade (`AsyncIterable<UserMessage>` as `prompt`), but it is **not** true bidirectional streaming.

**Date**: 2026-05-24
**Spike author**: Claude (delegated by Phase 5.S2 issue #63)
**Refines**: Phase 5.3 (Streaming input mode)

## TL;DR

- `callModel(client, request)` is single-prompt-per-invocation. `request.input` is `Item[] | string`, evaluated once at construction; the returned `ModelResult` has `cancel()`, `getState()`, consume-streams, and approval methods — **no `push` / `send` / `input` method.**
- The SDK already implements `interruptedBy` polling in `StateAccessor` (`model-result.js:316` `checkForInterruption`, `:798` resume path). External code writes `interruptedBy: "<reason>"` to state → the running loop saves partial state and exits cleanly → next `callModel` resumes from saved `messages` with the flag cleared.
- OR's wire transport is SSE (`@openrouter/sdk/esm/funcs/betaResponsesSend.js:28`), which is unidirectional by spec — no bidirectional duplex available at the HTTP layer.
- Claude Agent SDK's streaming input uses `prompt: AsyncIterable<SDKUserMessage>` plus a `.interrupt()` method on the `Query` controller; it is a *closed-loop* SDK pattern, not an HTTP-level duplex pattern, so it can be approximated on top of OR's interrupt-then-restart primitive.
- **Recommendation**: build Card 5.3 as a thin facade — `streamingInput(): AsyncIterable<UserMessage>` accepted on `OpenRouterAgentRun` (or top-level `query`-equivalent) — wired to interrupt-then-restart. Estimate refined to **16–24h** (vs. original 10–20h) because of the state-merge + partial-response handling.

## SDK surface findings

### `@openrouter/agent` `callModel`

SDK version inspected: **`@openrouter/agent@0.3.2`** (`package.json:35`, lockfile pinned).

**Argument shape** (`@openrouter/agent/esm/inner-loop/call-model.d.ts:64`):

```ts
export declare function callModel<TTools, TSharedSchema, TShared>(
  client: OpenRouterCore,
  request: CallModelInput<TTools, TShared> & { sharedContextSchema?: TSharedSchema },
  options?: RequestOptions,
): ModelResult<TTools, TShared>;
```

`CallModelInput.input` (`esm/lib/async-params.d.ts:17`):

```ts
input: FieldOrAsyncFunction<Item[]> | string;
```

`input` resolves **once per turn** via `resolveAsyncFunctions(...)`. While it can be a function (recomputed each turn), the function only receives `TurnContext` (turn number, etc.) — it has no access to a host-pushed message queue, no `AsyncIterable`-typed variant, and is awaited synchronously inside the loop. Treating `input` as a generator that blocks for new host input is not supported by the loop scheduler.

`grep -rE "push|inject|send.*[Mm]essage|input.*[Mm]essage|appendUserMessage"` against `node_modules/@openrouter/agent/esm/` returns zero matches on `ModelResult`. There is no host-side write path into a running call.

### The `interruptedBy` mechanism (the workaround foothold)

The SDK *does* expose a clean interrupt-then-resume primitive via `StateAccessor`:

`@openrouter/agent/esm/lib/tool-types.d.ts:555–557`

```ts
/** Signal from a new request to interrupt this conversation */
interruptedBy?: string;
/** Current status of the conversation */
status: ConversationStatus; // 'complete' | 'interrupted' | 'awaiting_approval' | 'in_progress'
```

`@openrouter/agent/esm/lib/model-result.js:316` — `checkForInterruption(currentResponse)`:

```js
async checkForInterruption(currentResponse) {
    if (!this.stateAccessor) return false;
    const freshState = await this.stateAccessor.load();
    if (!freshState?.interruptedBy) return false;
    // Save partial state (tool calls + partial response text + status='interrupted')
    // …
    return true;
}
```

Called on every loop iteration (`model-result.js:1058`):

```js
while (true) {
    if (await this.checkForInterruption(currentResponse)) {
        return; // exit cleanly with state persisted as 'interrupted'
    }
    if (await this.shouldStopExecution()) break;
    // … make next turn’s request
}
```

Resume path (`model-result.js:798`):

```js
if (loadedState.interruptedBy) {
    // Clear flag, set status='in_progress', save, continue with fresh callModel input
    this.currentState = updateState(loadedState, { status: 'in_progress' });
    this.clearOptionalStateProperties(['interruptedBy']);
    await this.saveStateSafely();
}
```

So the externally-visible pattern is: **(host) state.update({ interruptedBy: 'user-pushed-msg' }) → (loop) detects on next iteration → state goes to `'interrupted'` with partial response captured → (host) starts new `callModel` with appended user message in the `input`; the loaded state's prior `messages` are preserved as history, and the new `input` is the next user turn.**

Important caveats observed in `model-result.js`:

- Polling is only between turns / between SSE event batches — there is no preemption inside a single SSE token stream. A long single-response generation cannot be cut short until the API emits its end-of-response event, *or* the consumer calls `result.cancel()` (which tears the SSE stream down hard and leaves state in whatever the last persisted snapshot was).
- `cancel()` is the harder primitive — it kills the SSE reader but does not persist partial assistant text or partial tool calls. The "graceful" mid-stream stop is `interruptedBy`, but it only fires *between* loop iterations.
- For true mid-token interruption we'd need to call both `cancel()` (to stop the wire) and rebuild state ourselves from whatever events the consumer has seen — which our `for await … getFullResponsesStream()` already produces, so it is technically tractable.

### OR HTTP transport

`@openrouter/sdk/esm/funcs/betaResponsesSend.js:28`:

```js
"Accept": stream === true ? "text/event-stream" : "application/json",
```

`@openrouter/sdk/esm/lib/sdks.js:253`:

```js
case matchContentType(res, "text/event-stream"):
    // SSE decode path
```

OR's beta `/responses` endpoint (which `callModel` targets) uses **SSE** for streaming. SSE is unidirectional by RFC; the client cannot send body chunks after the initial POST. No WebSocket or HTTP/2 duplex endpoint is exposed on OpenRouter as of v0.12.35 (`@openrouter/sdk@0.12.35`). Therefore **true wire-level streaming input is impossible** against the current OR API — even if `callModel` exposed a `push()` method, it would internally have to either restart the SSE call or hold a side channel that does not exist.

## Reference implementations

### Claude Agent SDK streaming input

Claude Code's `@anthropic-ai/claude-agent-sdk` (Anthropic's official SDK, public on npm) shape (verified against the public TS surface):

```ts
function query(options: {
  prompt: string | AsyncIterable<SDKUserMessage>;
  options?: Options;
}): Query;

interface Query extends AsyncIterable<SDKMessage> {
  interrupt(): Promise<void>;
  setPermissionMode(mode: PermissionMode): Promise<void>;
  // …
}

interface SDKUserMessage {
  type: 'user';
  message: { role: 'user'; content: string | ContentBlock[] };
  // …
}
```

The streaming-input contract:

1. Host passes `prompt: AsyncIterable<SDKUserMessage>`. The SDK awaits the first message to start the first turn.
2. After the turn ends, the SDK pulls the *next* yielded message from the same iterator and submits it as the next user turn (with full history preserved on the SDK side).
3. `query.interrupt()` is a side-channel control message that aborts whatever the model is currently doing, including mid-tool-call, and waits for the SDK to land in a clean state before returning.
4. Image attachments and other content blocks ride inside the same `ContentBlock[]` array on the yielded `SDKUserMessage`.

This is **not** an HTTP-level duplex; it's a long-lived in-process loop driven by the host's iterator. Each "push" translates to a new chat-completions request with the accumulated messages array — Anthropic's wire is also SSE. The streaming-input *ergonomic* is purely SDK-side glue around `AsyncIterable` + `interrupt()`.

That matters here because it means **we can ship the same ergonomic on top of OR's interrupt-then-restart primitive**, with the only behavioral gap being: `query.interrupt()` on Claude has well-defined partial-response handling (the partial assistant message is committed and the interrupt is its own message-type in the transcript). On OR, our `interruptedBy` flag does persist the partial response into `ConversationState.partialResponse`, so the equivalent commit semantics are reachable.

Sources: `@anthropic-ai/claude-agent-sdk` public README + the in-tree `Query` interface (Anthropic's claude-code SDK ships its TypeScript types in the npm package, e.g. `dist/index.d.ts`).

### opencode

opencode (`https://github.com/sst/opencode`) is a Claude-Code-like multi-provider CLI. Inspected for any streaming-input bridging for non-Anthropic providers:

- opencode currently models user input as discrete prompts per "send" — its UI fires one `prompt()` call per submission and that maps to one provider call. There is no `AsyncIterable<UserMessage>` surface for any provider it bridges.
- opencode handles interrupts via the standard AbortSignal pattern over its provider clients (Anthropic, OpenAI, OR, etc.) — i.e. cancel-and-stop, with the user free to send a new message that becomes a new turn. The user-facing UX is the workaround pattern.

So opencode itself does **not** offer a richer-than-OR pattern here; it sits at the "cancel + restart" tier, which is the same tier we'd land at. That's useful confirmation that the Claude-Code streaming-input ergonomic is currently *unique to Anthropic's SDK* and is not yet expected behavior across the Claude-Code-clone ecosystem.

## Verdict + Path forward

### If Yes — N/A (not supported by SDK or wire transport)

Skipped.

### If No (the verdict) — workaround design sketch

**Workaround**: interrupt-then-restart with appended prompt, hidden under a Claude-SDK-shaped facade so that consumers can write the same `AsyncIterable<UserMessage>` pattern they would against Anthropic's SDK.

#### API shape (proposed for Card 5.3)

```ts
// Top-level surface mirroring Claude SDK
interface OpenRouterAgentRun extends AsyncIterable<AgentEvent> {
  // Existing:
  abort(reason?: unknown): void;

  // New (Card 5.3):
  pushUserMessage(msg: UserMessage | string): Promise<void>;
  interrupt(): Promise<void>;
}

function run(opts: {
  prompt: string | AsyncIterable<UserMessage>; // ← AsyncIterable enables streaming input
  // …existing opts
}): OpenRouterAgentRun;
```

Internal flow:

1. Outer loop owns a `StateAccessor` and drives a sequence of `callModel(…)` invocations, one per user turn.
2. `pushUserMessage(msg)` enqueues into an internal `messageQueue: UserMessage[]`.
3. Between turns, the outer loop pulls the next message off the queue (or from the consumer's `AsyncIterable`) and supplies it as `input` to the next `callModel` call. History is carried via `StateAccessor.messages` — already shipping per Spike 5.S1.
4. `interrupt()` writes `state.interruptedBy = '<host-interrupt>'` and awaits the current `callModel` promise to resolve (the loop will observe the flag on next iteration and exit cleanly with `status: 'interrupted'` + `partialResponse` populated).
5. The next `pushUserMessage` starts a new `callModel` with the prior `messages` + `partialResponse` (committed as an interrupted assistant turn) + the new user message as `input`.

#### Trade-offs vs. Claude SDK's streaming input

| Property                                | Claude SDK                                      | This workaround on OR                                                                     |
| --------------------------------------- | ----------------------------------------------- | ----------------------------------------------------------------------------------------- |
| Push new user msg mid-turn              | Buffered; appended to next turn                 | Same: buffered, appended to next turn                                                     |
| Interrupt mid-token                     | Yes, clean transcript commit                    | Possible but coarser — `cancel()` cuts SSE hard; `interruptedBy` only fires between turns |
| Image attachments                       | Native `ContentBlock` support                   | Native — `UserMessage.content` already accepts `ContentBlock[]` via OR Responses API      |
| Tool-call interrupt                     | Tool exec is cancelable mid-flight via signal   | Tool exec can be canceled via existing `AbortController` pipeline                         |
| State after interrupt                   | Partial assistant message committed             | Partial assistant text persists in `ConversationState.partialResponse`                    |
| Round-trips                             | 1 long-lived loop, N turns                      | N separate `callModel` HTTP requests (one per user push); no extra cost                   |
| Latency on push                         | "Instant" within same loop                      | One extra request boundary per push (cold start = ~one HTTP round-trip)                   |

The main observable difference is the **mid-token interrupt granularity**. For our typical agent loop (which is mostly tool-call-bound, not text-streaming-bound), this is acceptable — interrupt requests will land between turn boundaries within ~the duration of one tool call, which matches the user's mental model of "I told it to stop; it stopped after the current step."

#### Implementation cost breakdown

- AsyncIterable-shaped `prompt` adapter (parse first message, expose pull-next): **2–3h**
- `messageQueue` + `pushUserMessage` plumbing on `OpenRouterAgentRun`: **3–4h**
- `interrupt()` wired to `StateAccessor.update({ interruptedBy })` + await-current-call: **3–4h**
- Multi-turn restart loop (resume from prior `messages` + commit `partialResponse`): **4–6h**
- Tests (queue ordering, interrupt-between-turns, interrupt-mid-tool-call, image attachment passthrough, partial-response commit): **4–6h**

**Total refined: 16–24h** (vs. roadmap placeholder 10–20h; the bump is the partial-response commit + the multi-turn restart loop, both of which are easy to underestimate).
