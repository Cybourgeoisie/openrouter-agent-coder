# Spike 5.S1 — Does `@openrouter/agent` expose message history?

> Issue: [#62](https://github.com/Cybourgeoisie/openrouter-agent-coder/issues/62) — investigation only, no `src/` changes.

## 1. Verdict

**Yes — fully exposed.** The SDK persists the entire turn-by-turn message history (user, assistant, tool calls, tool outputs, reasoning items) on `ConversationState.messages`, which our existing `StateAccessor` already round-trips through `state.json`. Compaction is a straight read-mutate-write against that array — no SDK changes or extra endpoints needed.

## 2. Evidence

All file paths are relative to `node_modules/`. SDK version pinned: `@openrouter/agent@^0.3.2` (`package.json:35`).

### 2a. `ConversationState.messages` holds the full history

`@openrouter/agent/esm/lib/tool-types.d.ts:542`

```ts

  id: string;
  /** Full message history */
  messages: models.InputsUnion; // ← the entire transcript
  previousResponseId?: string; // ← also present, but NOT the only history mechanism
  pendingToolCalls?: Array<ParsedToolCall<TTools[number]>>;
  unsentToolResults?: Array<UnsentToolResult<TTools>>;
  partialResponse?: PartialResponse<TTools>;
```

### 2b. The SDK actively appends to `messages` every turn

`@openrouter/agent/esm/lib/model-result.js:282` (assistant response items):

```js
messages: appendToMessages(this.currentState.messages, outputItems),
```

`@openrouter/agent/esm/lib/model-result.js:306` (tool results, next turn):

```js
messages: appendToMessages(this.currentState.messages, toolResults),
```

And on resume `@openrouter/agent/esm/lib/model-result.js:843`:

```js
input: appendToMessages(this.currentState.messages, inputArray),
```

So the SDK both **writes** the full transcript into `state.messages` and **rehydrates** it as the next `callModel` `input` when state is loaded — meaning if we mutate `messages` before the next turn (compaction), the SDK will pick up the rewritten history transparently.

### 2c. Public APIs to read/write history

`@openrouter/agent/esm/index.d.ts` exports both halves of the contract:

- `appendToMessages(current, newItems)` — public helper for splicing items.
- `ModelResult.getState()` — public, returns `Promise<ConversationState<TTools>>` for inspection (`model-result.d.ts`: _"Get the current conversation state. Useful for inspection, debugging, or custom persistence."_). Impl at `model-result.js:1522`.
- `StateAccessor` interface (`tool-types.d.ts:570`) — the existing load/save hook we already implement in `src/state/file-state.ts`. Our on-disk `state.json` already contains the full message history; the spike author confirmed this is the canonical persistence point the SDK reads from on resume.

### 2d. Not just `previousResponseId`

The issue framed this as a binary: history exposed _or_ OR treats chains opaquely via `previousResponseId`. The reality is both ship together — `previousResponseId` is a server-side optimization (`tool-types.d.ts:548` comment: _"OpenRouter server-side optimization"_), while `messages` is the authoritative client-side mirror. Compaction lives on the client side; `previousResponseId` is orthogonal (we may want to **clear it** when we compact, so the server doesn't try to splice a stale chain onto our rewritten input — see §3).

### 2e. OR docs cross-check

No `/v1/responses/<id>/messages` chain-traversal endpoint is needed or documented — the SDK's local `state.messages` is the source of truth. We never have to hit a remote endpoint to enumerate prior turns. (This is the same model Anthropic's SDK uses for Claude Code's `/compact`: it owns the full transcript locally and rewrites it before the next request.)

## 3. How compaction would consume this

Conceptually a `compactSession()` helper would, given an active `Agent`/session id:

1. **Load** current state via the existing `FileStateAccessor.load()` — gets `ConversationState` with the full `messages` array.
2. **Choose a cut point.** Simplest policy: keep the system message (if any), keep the latest N messages (e.g. last 6 turns), and summarize everything between. More refined policies can compute a token budget — see §5 assumption (a).
3. **Generate summary.** Issue a fresh `client.callModel({ input: <prefix to summarize>, ... })` with a "summarize prior conversation" instruction. Capture `getText()`. (We may need a side-channel session id so the summary call isn't itself persisted under the user's session.)
4. **Rewrite messages.** Build a new array: `[<optional system msg>, { role: 'developer' | 'user', content: <summary> }, ...<recent tail>]`.
5. **Clear chain pointer.** Drop `previousResponseId` from state — otherwise the server will try to splice the rewritten input onto a chain it no longer matches. Set `status: 'complete'`, refresh `updatedAt`.
6. **Save** via `FileStateAccessor.save(newState)`.
7. **Next `Agent.run()`** loads the compacted state and proceeds as normal — no SDK changes needed.

Sketch (illustrative — not real src changes):

```ts
const state = await stateAccessor.load();
if (!state || !Array.isArray(state.messages)) return;

const { keep, summarize } = partitionByTokenBudget(state.messages, budget);
const summaryText = await summarizer
  .callModel({
    sessionId: `${sessionId}-compact-${Date.now()}`,
    input: [
      {
        type: 'message',
        role: 'developer',
        content: 'Summarize the prior conversation concisely.',
      },
      ...summarize,
    ],
    // tools omitted — pure text task
  })
  .getText();

const compacted: ConversationState = {
  ...state,
  messages: [
    { type: 'message', role: 'developer', content: `[Compacted prior context]\n${summaryText}` },
    ...keep,
  ],
  previousResponseId: undefined,
  status: 'complete',
  updatedAt: Date.now(),
};
await stateAccessor.save(compacted);
```

The shape of an `InputMessageItem` / `EasyInputMessage` is fully typed (see `@openrouter/agent/esm/index.d.ts` re-exports) so we can construct the summary entry without guessing.

## 4. Open questions (do not block 5.1)

- **Token counting.** The SDK doesn't expose a tokenizer. Options for the budget decision: (a) character-length heuristic (~4 chars/token), (b) call OR's `/api/v1/responses` with `usage` and read `inputTokens` back, (c) ship `js-tiktoken` for cl100k-ish estimates. (a) is good enough for an MVP; (c) is the precise answer at the cost of one dep.
- **Summary persistence.** Do we want compaction events surfaced through `Agent.run()`'s message stream (per Card 3.8) so callers can render a "context compacted" notice? Probably yes — adds a small `compaction_complete` event type.
- **PreCompact hook.** Roadmap already calls this out under Card 5.1. Plumbing point would be right before step 3 above (so a hook author can veto/customize). Same `safeFireHook` pattern as Setup/PreToolUse.

## 5. Refined estimate for Card 5.1

**Previous placeholder:** 15–25h ("compaction"). **Refined:** **12–18h**. Slight reduction because:

- History access is _trivially solved_ (already exposed; we already persist it) — eliminates ~3h of plumbing risk that the original range padded for.
- Compaction itself decomposes cleanly: (a) tokenizer/budget — 2–3h, (b) cut-point selection + summary call — 3–5h, (c) state rewrite + `previousResponseId` clear — 1h, (d) `PreCompact` hook — 1h, (e) `compaction_complete` message-stream event + `aggregateMessages` integration — 1–2h, (f) tests (history fixture, integration end-to-end, hook contract) — 4–6h.

**Assumptions baked into the estimate:**

1. We adopt char-length heuristic for v1 token counting (defer `js-tiktoken` dep to a follow-up if it matters).
2. Summary call uses the same `OR_MODEL` as the main agent by default; configurable via `OR_COMPACT_MODEL`. No new model-routing logic.
3. `PreCompact` hook signature mirrors existing `safeFireHook` pattern (event name + sessionId/cwd payload + cancellable). No new hook infrastructure.
4. Tests use the existing `.test-tmp/` filesystem rig and the recorded-fixture pattern (`scripts/record-fixture.ts`) — no new test infra.
5. The compaction trigger (auto on N% of context vs. explicit `/compact` command vs. both) is decided as part of 5.1 scoping, not redesigned here. Estimate assumes auto-trigger with a manual override path.

If assumption (1) is rejected and we ship `js-tiktoken` properly with model-aware encodings, add ~3h.
