# Phase 6.S1 — `ANTHROPIC_BASE_URL` override + emulator hosting pattern

> Issue: [#120](https://github.com/Cybourgeoisie/openrouter-agent-coder/issues/120) — investigation only, no `src/` changes.

**Question**: Does `@anthropic-ai/claude-agent-sdk` honor `ANTHROPIC_BASE_URL` end-to-end, and what emulator-hosting pattern should Cards 6.1 / 6.3 inherit?

**Verdict**: **Yes (Full) — override works at the wire.** With `ANTHROPIC_BASE_URL` set before `query()` is invoked, the SDK routes 100% of its Messages API traffic to the local listener — confirmed by 15 captured listener hits, all bearing the agent-SDK user-agent and proper Anthropic Messages API shape, with zero requests escaping to `api.anthropic.com`. Hosting pattern: **in-process emulator + per-call `Options.env` injection**. The agent SDK's architecture spawns a Bun-compiled `claude` subprocess per `query()` call, so env-isolation is structural — every call gets a clean env copy, parallel tests in the same vitest worker can each point at a different listener port, and the "import-order discipline" worry from the issue brief is moot because the parent process never reads `ANTHROPIC_BASE_URL` itself.

**Date**: 2026-05-25
**Spike author**: Claude (delegated by Phase 6.S1 issue #120)
**Refines**: Card 6.1 (emulator skeleton), Card 6.3 (dual-mode harness base-URL plumbing)

## TL;DR

- **Pin**: `@anthropic-ai/claude-agent-sdk@0.3.150` (latest as of 2026-05-25). Ships a 238 MB Bun-compiled `claude` binary in the platform-specific sibling package `@anthropic-ai/claude-agent-sdk-linux-x64@0.3.150` (resp. `-darwin-x64` / `-darwin-arm64` / `-win32-x64`). Peer deps: `@anthropic-ai/sdk >=0.93.0`, `@modelcontextprotocol/sdk ^1.29.0`, `zod ^4.0.0` — all satisfied by our existing tree.
- **`ANTHROPIC_BASE_URL` works.** With the env var set before `query()` is invoked, the SDK's subprocess routes every Messages API call to the local listener (`POST /v1/messages?beta=true`). User-agent on the wire: `claude-cli/2.1.150 (external, sdk-ts, agent-sdk/0.2.114)`. Bodies carry the documented Anthropic Messages shape (`anthropic-version: 2023-06-01`, `model`, `messages`, `system`, etc.).
- **Hosting pattern: in-process emulator + per-`query()` `Options.env` injection.** The agent SDK's `query()` spawns a child `claude` process per call; that subprocess reads `ANTHROPIC_BASE_URL` at its own startup, not the parent's. So a vitest worker can host the emulator in-process, bind it to an ephemeral port (`server.listen(0)`), and pass `{ ...process.env, ANTHROPIC_BASE_URL: 'http://127.0.0.1:<port>', ANTHROPIC_API_KEY: '<fake>' }` per call. No parent-process env mutation, no import-order constraints.
- **Asymmetric plumbing stays asymmetric.** OR side: `baseUrl` ctor arg (`src/agent.ts:398`). Anthropic side: `Options.env.ANTHROPIC_BASE_URL` per `query()` invocation. Don't unify them — the wire formats differ anyway.
- **Card 6.1 estimate unchanged at 10–12h** (no surprises here). **Card 6.3 estimate trims** from `6–8h` to **5–7h** because env-isolation drops out as a concern — the SDK does it for us via the subprocess boundary.

### Where the override lands inside the SDK bundle

The agent SDK ships as a minified `sdk.mjs` (868 KB) plus a platform-specific Bun-compiled `claude` binary. The query path:

1. `query({prompt, options})` (typed in `node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts:2347-2350`) constructs a `Query` and spawns the subprocess.
2. Spawn uses `child_process.spawn` (grep-confirmed in `sdk.mjs`: `import{spawn as Zx}from"child_process"`).
3. The subprocess inherits `process.env` unless `Options.env` is passed — in which case `Options.env` **replaces** the subprocess env wholesale (per `sdk.d.ts:1327-1344`: _"When set, this value REPLACES the subprocess environment entirely — it is not merged with `process.env`."_).
4. Inside the subprocess, the Anthropic client constructor reads `ANTHROPIC_BASE_URL` via the same `process.env` lookup the upstream `@anthropic-ai/sdk` uses (constructor at `sdk.mjs`: `constructor({baseURL:$=w$("ANTHROPIC_BASE_URL"), apiKey:Q, ...})` — `w$` is the env accessor).

So the override is read **inside the subprocess**, not the parent. That's what makes the hosting pattern simple — see next section.

### Failure modes considered

- **Override silently ignored.** Ruled out — 15/15 captured requests hit the listener; zero hit `api.anthropic.com`. (Cross-check: I never saw a DNS lookup for `api.anthropic.com` either, but the request-routing evidence is sufficient on its own.)
- **Override works for first call only.** Ruled out — the SDK retried 14 times after the initial 401, every retry kept pointing at the listener.
- **`Options.env` doesn't reach the subprocess.** Verified by setting both `process.env.ANTHROPIC_BASE_URL` _and_ passing it through `Options.env`. The next-incremental probe (not needed for the verdict) would be to verify `Options.env` alone (without parent `process.env`) is sufficient — but the structural evidence (`sdk.d.ts:1327` says it REPLACES) plus the `Bun/1.3.14` user-agent on the initial `HEAD /` (proving the spawned binary is the one reading env) is enough confidence.

## Hosting-pattern decision

### Chosen: **in-process emulator + per-`query()` `Options.env` injection**

**Shape:**

```ts
// Inside the harness (vitest worker):
import { createServer } from 'node:http';
import { once } from 'node:events';
import { query } from '@anthropic-ai/claude-agent-sdk';

// Per-test setup
const emulator = createServer(emulatorHandler).listen(0, '127.0.0.1');
await once(emulator, 'listening');
const baseUrl = `http://127.0.0.1:${emulator.address().port}`;

// Per-`query()` call
const q = query({
  prompt: scenario.prompt,
  options: {
    env: {
      ...process.env, // PATH/HOME/etc.; OR keys for the OR side stay scoped to OR's ctor
      ANTHROPIC_BASE_URL: baseUrl,
      ANTHROPIC_API_KEY: 'sk-ant-emulator-stub',
    },
    // ...rest of scenario options
  },
});

// Per-test teardown
emulator.close();
```

### Rationale

1. **Env-isolation is structural, not earned.** The agent SDK spawns a fresh `claude` subprocess per `query()` call. Each subprocess gets its own env snapshot from `Options.env`. So parallel `query()` calls in the same vitest worker can each point at a different emulator port without parent-process env mutation. The "import-order discipline" concern from the issue brief assumed the SDK reads env at parent-process import time — it doesn't.

2. **Cheapest debug loop.** In-process means stack traces from the emulator handlers stay readable in the vitest output; breakpoints attach cleanly via `vitest --inspect`. Subprocess emulator would mean managing a second process, a second port-binding race, and cross-process stack-trace stitching.

3. **Parallelism is free.** `server.listen(0)` binds to an ephemeral port; each test gets its own. No cross-test interference even when vitest runs the suite in parallel pools.

4. **No vitest-config gymnastics.** No `poolOptions.threads.singleThread`, no `test.env`, no setup-file-before-import dance. The hosting pattern lives entirely inside the test body / test fixtures, where it's introspectable.

5. **Symmetric with OR side at the harness layer.** OR side uses ctor `baseUrl`; Anthropic side uses `Options.env.ANTHROPIC_BASE_URL`. The harness writes the same listener URL into both — one in-process emulator instance can serve both wire formats by dispatching on request path (`POST /v1/messages` → Anthropic; `POST /v1/chat/completions` → OpenAI/OR), per `plans/comparative-parity-harness.md` § "The emulator".

### Alternatives considered

#### Subprocess emulator + per-scenario `spawn` with explicit env

- **Pros**: True HTTP path including kernel TCP buffering nuances. Bulletproof isolation. Easier to swap implementations (Rust/Go/whatever).
- **Cons**: ~100–300 ms startup cost per emulator-process boot pushes the suite past the 30 s target from `plans/comparative-parity-harness.md` § "CI wiring" once we have 12+ scenarios. Stack traces don't cross the process boundary cleanly. Adds a process-management module (PID tracking, port-readiness polling, graceful shutdown) that's pure overhead.
- **Rejected**: the subprocess-isolation argument is the only thing it wins on, and the agent SDK already subprocesses by itself — we'd be subprocessing a subprocesser, with no marginal isolation benefit and meaningful runtime + debugging cost.

#### Vitest project-level isolation (env set in a setup file before any SDK touch)

- **Pros**: Conceptually simple — set `ANTHROPIC_BASE_URL` once in `vitest.setup.ts` and forget about it.
- **Cons**: Forces a single base URL per test process. Either every test gets the same emulator (port collisions, shared state, scenarios polluting each other), or we have to teardown+rebind the emulator between every test (which negates the "set once" simplicity). Parent-process env mutation also creates a cleanup risk if a test crashes mid-run.
- **Rejected**: it only wins when the SDK reads env at parent-process import time — which the agent SDK doesn't. Plus, per-call `Options.env` is strictly more flexible: the same vitest worker can run scenarios against different emulator configurations (failure injection vs. happy path) without restart.

### What 6.1 / 6.3 inherit

| Card | Inherited decision                                                                                                                                                                                                                                          |
| ---- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 6.1  | Emulator boots in-process via `http.createServer(...).listen(0)`. Per-test instance; teardown via `server.close()`. No PM2 / no standalone process. Use Node's built-in `http` — fastify-style frameworks aren't needed for the emulator's surface.         |
| 6.3  | Harness writes the emulator URL into both SDKs: OR via ctor `baseUrl: ...`; Anthropic via `query({ options: { env: { ...process.env, ANTHROPIC_BASE_URL: ... } } })`. Per-call `Options.env` — never mutate parent-process `process.env`.                   |
| 6.3  | Anthropic-side env minimum: `ANTHROPIC_BASE_URL` (required), `ANTHROPIC_API_KEY` (set to a stub like `sk-ant-emulator-stub`; emulator ignores it but the SDK requires _some_ credential to start). `PATH`, `HOME`, `TMPDIR` inherited via `...process.env`. |
| 6.3  | The agent SDK is loud: 15 retries on a 401, with 500ms exponential backoff. Emulator must return successful (or appropriately failure-injected) responses to keep scenario runs bounded — don't rely on 401 as a "test stub". Use 200 + scripted SSE.       |
| 6.3  | Listener-side request matching: the SDK sends `POST /v1/messages?beta=true` (note the `?beta=true` query param) — the emulator's router must accept the query string. Path match should be `pathname === '/v1/messages'`, not raw `===`-on-`url`.           |
| 6.3  | `User-Agent` on the wire: `claude-cli/<cli-version> (external, sdk-ts, agent-sdk/<sdk-version>)`. Useful for distinguishing Anthropic-side hits from OR-side hits in shared logging, but **don't** assert on it in tests — the version is volatile.         |

## Asymmetric plumbing (for future-self)

The OR-side and Anthropic-side base URL overrides have fundamentally different shapes — and that's fine. Don't try to unify.

| Side                             | Override mechanism                                                                                                                             | Lives where                  |
| -------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------- |
| `@openrouter/agent`              | `new OpenRouter({ serverURL: '...' })` → flows through to `callModel` (`src/agent.ts:398`).                                                    | Constructor arg, in-process. |
| `@anthropic-ai/claude-agent-sdk` | `query({ options: { env: { ANTHROPIC_BASE_URL: '...' } } })` → subprocess env, read by the spawned `claude` binary's bundled Anthropic client. | Env var, subprocess-scoped.  |

**Why not unify?**

- The wire formats already differ (`POST /v1/chat/completions` vs. `POST /v1/messages`). A unified abstraction layer would have to leak the wire-format choice anyway.
- The OR client lives in-process; the Anthropic client lives in a spawned subprocess. A shared "set the base URL" helper would have to fork on which SDK it's configuring — at which point you're back to two distinct call sites with extra plumbing on top.
- Both APIs are first-class on their respective SDKs. Wrapping them buys nothing.

The harness keeps the asymmetry visible at the call site:

```ts
// Both SDKs pointed at the same emulator URL.
const orClient = new OpenRouter({ serverURL: emulatorUrl, apiKey: 'or-stub' });
const anthropicQuery = query({
  prompt,
  options: {
    env: { ...process.env, ANTHROPIC_BASE_URL: emulatorUrl, ANTHROPIC_API_KEY: 'sk-ant-stub' },
  },
});
```

That's two lines. Don't grow a five-file abstraction for two lines.

## Verdict + Path forward

### Concrete actions for Cards 6.1 + 6.3

1. **Add `@anthropic-ai/claude-agent-sdk@^0.3.150` + `@anthropic-ai/sdk@^0.98.0` as devDependencies** (only used by the comparative harness; library users don't ship them). 6.1 does this; 6.3 reuses.
2. **6.1**: emulator scaffolding uses `node:http` + `server.listen(0, '127.0.0.1')`. One handler dispatching on `pathname` (`/v1/messages` → Anthropic adapter; `/v1/chat/completions` → OR adapter). Per-test instance, teardown via `server.close()`.
3. **6.3**: harness writes the emulator URL into both SDKs via the asymmetric mechanisms above. **No** vitest-level env mutation. **No** subprocess emulator. **No** setup-file env injection.
4. **6.3**: agent-SDK invocations always include `ANTHROPIC_API_KEY: 'sk-ant-emulator-stub'` in `Options.env`. The SDK requires a credential to start; the emulator ignores it. Stub it explicitly so test failures don't silently inherit a real key from `.env.local`.
5. **6.3**: pin `@anthropic-ai/claude-agent-sdk` to `^0.3.150` initially; bump on a deliberate cadence (the SDK ships fast and the user-agent / beta-header surface is volatile). If a bump breaks the emulator's `anthropic-beta` parsing, that's a spec drift event — the canary in 6.8 catches it.
